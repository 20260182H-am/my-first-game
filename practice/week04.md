## AI 대화 기록

 ### 1.AABB에서는 sqrt를 쓰지 않고 circle에서는 sqrt를 쓰는 이유는 무었인가?

AI 답변: AABB(축 정렬 바운딩 박스)는 x축과 y축을 기준으로 겹치는지만 확인하면 되기 때문에 단순한 비교 연산만으로 충돌을 판별할 수 있다. 즉, 두 사각형의 좌우, 상하 범위를 비교하면 되므로 거리 계산이 필요 없다.
반면 circle(원형 충돌)은 두 원의 중심 사이의 거리를 계산해야 한다. 이 거리가 두 원의 반지름의 합보다 작으면 충돌로 판단한다. 이때 중심 간의 실제 거리를 구하기 위해 피타고라스 공식을 사용하며, 이 과정에서 sqrt(루트)가 필요하다.

AABB 충돌
if rect1.right > rect2.left and rect1.left < rect2.right:
    if rect1.bottom > rect2.top and rect1.top < rect2.bottom:
        print("충돌!")

Circle 충돌
dx = x1 - x2
dy = y1 - y2
distance = (dx**2 + dy**2) ** 0.5

if distance < r1 + r2:
    print("충돌!")

적용 결과: AABB는 빠르게 충돌을 판별할 수 있었고, circle은 sqrt 연산이 들어가지만 더 정확한 원형 충돌 판정이 가능했다.


### 2. OBB를 쓰는 데 SAT 방식이 필수인가? 회전하는 방향에 맞게 바운딩박스를 돌리고, 닿았을 때 감지하는 방식은 안되는가?

AI 답변: 단순히 박스를 회전시키는 것만으로는 정확한 충돌 판정이 어렵다. OBB(Oriented Bounding Box)는 회전된 사각형이기 때문에 AABB처럼 단순 비교로는 겹침 여부를 알 수 없다.
그래서 SAT를 사용하여 값을 구한다. 이 방법은 두 도형을 특정 축에 투영해서 겹치는지 확인하는 방식이다. 만약 하나라도 겹치지 않는 축이 있으면 충돌하지 않은 것이다.
회전된 박스를 만들고 닿았을 때 감지하는 방식은 결국 내부적으로 SAT 같은 계산을 해야 정확한 결과가 나오기 때문에 단순 좌표 비교만으로는 회전된 모양을 처리하기 어렵다.

SAT에서 사용하는 축 생성 (간단 개념)
axes = [edge1_normal, edge2_normal]

for axis in axes:
    proj1 = project(rect1, axis)
    proj2 = project(rect2, axis)

    if not overlap(proj1, proj2):
        print("충돌 없음")
        break

적용 결과: 단순 회전 박스 비교는 부정확했지만, SAT 방식을 적용하자 회전된 물체끼리도 정확하게 충돌을 판정할 수 있었다.

### 3. AABB, circle, OBB 방식에는 각각 어떤 장, 단점이 있는가

AI 답변:

AABB (축 정렬 박스)
장점: 계산이 매우 빠름 (단순 비교), 구현이 쉬움
단점:회전된 물체에 부정확함, 실제 모양보다 충돌 범위가 큼

Circle (원형 충돌)
장점: 방향에 상관없이 항상 정확함, 구현이 비교적 간단함
단점: sqrt 연산이 필요해서 AABB보다 느림, 사각형 모양과는 맞지 않을 수 있음

OBB (회전 박스)
장점: 실제 물체 모양에 더 가깝게 충돌 판정 가능, 회전 대응 가능
단점: 계산이 복잡함 (SAT 필요), 구현 난이도가 높음


## 관찰 내용

### 원형 VS AABB(모서리 부분에 어떤 차이가 있는가?)
원형의 경우, 두 오브젝트가 모서리 쪽에서 살짝 닿았을 때 충돌이 감지되지 않을 수 있다.
반면 AABB는 사각형 기준으로 범위를 체크하기 때문에, 모서리 부분에서도 충돌이 감지된다.

### AABB VS OBB(회전 시 어떤 차이가 있는가?)
AABB는 오브젝트가 회전하더라도 충돌 범위가 항상 축에 맞춰 고정되어 있다.
반면 OBB는 오브젝트의 회전에 따라 바운딩 박스도 같이 회전하기 때문에, 회전이 있는 상황에서 더 정확하게 충돌을 감지할 수 있다.

### 세 가지 방식 중 어떤 것이 게임에 가장 적합한가?
가장 정확한 방식은 OBB라고 생각한다.
OBB는 오브젝트의 방향과 움직임(벡터)에 맞춰 충돌을 계산할 수 있기 때문에 회전이나 복잡한 움직임이 있는 게임에서 더 적합하다
하지만 캐주얼한 게임에서는 AABB나 circle의 활용도 역시 높을 것이다.

## AI와의 대화에서 배운 점

원형 충돌에서 sqrt 없이 계산할 수 있는 방법을 질문했는데, AI는 거리의 제곱을 이용하면 된다고 설명해 주었다.

def circle_collision(center1, r1, center2, r2):
 dx = center1[0] - center2[0]
 dy = center1[1] - center2[1] 
 
distance_sq = dx*dx + dy*dy 
radius_sum = r1 + r2 

return distance_sq <= radius_sum * radius_sum

방법이 궁금해서 질문했는데, AI는 거리의 제곱을 이용하면 된다고 설명해 주었다. 두 원의 중심 거리와 반지름의 합을 비교할 때, 
굳이 제곱근(sqrt)으로 실제 거리를 구하지 않고, 양쪽 값을 모두 제곱해서 비교해도 결과는 같다는 점을 들어 설명해 주었다.
특히 “키를 비교할 때 굳이 루트를 씌우지 않아도 된다”는 비유를 통해, (거리의 제곱)과 (기준의 제곱)을 비교하면 된다는 설명이 매우 이해하기 쉬웠다. 이 덕분에 왜 sqrt 없이도 충돌 판정이 가능한지 확실하게 이해할 수 있었다.
이전에 같은 내용을 설명해 달라고 했을 때, “수학적으로 이해할 수 있는, 이론 학습이 가능한”과 같은 식으로 질문하면 오히려 이해하기 어려웠고, 중학생 수준이나 비유를 활용한 설명을 요청했을 때 훨씬 잘 이해된다는 것을 알게 되었다.
이를 통해, 코딩을 처음 배우는 단계에서는 어려운 이론 중심의 설명보다, 일상적인 비유나 쉬운 표현을 활용한 방식으로 질문하는 것이 더 효과적일 수 있겠다는 생각이 들었다.

## 내 게임에 적용한다면?

나는 OBB 방식을 사용할 것 같다.
개인적으로 언리얼 엔진 기반의 게임이나 AAA급 게임을 좋아하기 때문에, 더 자연스럽고 정확한 충돌 판정이 중요하다고 생각한다. 이런 관점에서 보면 circle이나 AABB 방식은 상황에 따라 부자연스럽게 느껴질 수 있고, 게임의 몰입을 방해할 수도 있다고 느꼈다.
하지만 현재 나는 코딩을 처음 배우는 단계이고, 학교에서도 인디 형태의 게임을 만드는 상황이기 때문에 무조건 OBB만 고집하는 것은 비효율적일 수 있다. OBB는 정확하지만 계산이 복잡하고 구현 난이도가 높기 때문이다.
그래서 게임의 상황에 따라 방식을 유동적으로 선택하는 것이 가장 좋다고 생각한다. 예를 들어, 단순한 충돌은 AABB나 circle을 사용하고, 더 정밀한 판정이 필요한 부분에만 OBB를 적용하는 방식이 현실적이라고 느꼈다.
